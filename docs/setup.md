# Установка: шаг за шагом

Весь стек живёт в Docker Compose и поднимается одной командой. Ниже — что
нужно, в каком порядке и зачем каждый шаг

## Что понадобится

| Инструмент | Зачем |
| --- | --- |
| Docker + Docker Compose | запуск всего стека |
| Токен бота от @BotFather | демон живых обновлений (`bot`) входит в стек и не стартует без токена |
| `api_id` / `api_hash` с [my.telegram.org](https://my.telegram.org) | только для импорта истории канала (`make sync`) — можно отложить |
| Ключ [OpenRouter](https://openrouter.ai/keys) | только для генерации SEO-метаданных — можно отложить |

Go и Bun для запуска через Docker не нужны — они нужны только для
[локальной разработки](development.md)

## 1. Настроить окружение

```bash
cp .env.example .env
```

Минимум для первого запуска — токен бота:

1. Создайте бота у [@BotFather](https://t.me/BotFather)
2. Добавьте его администратором канала — хватает права «Изменение публикаций»
3. Пропишите токен в `.env`: `TELEGRAM_BOT_TOKEN=...`

Остальные переменные по группам:

| Группа | Переменные | Зачем |
| --- | --- | --- |
| База | `POSTGRES_USER/PASSWORD/DB`, `DATABASE_URL` | Postgres; дефолты годятся для локального запуска |
| Медиа | `MINIO_*`, `MEDIA_BASE_URL` | MinIO хранит фото и видео постов; `MEDIA_BASE_URL` — публичный адрес, по которому их видит браузер |
| Канал | `TELEGRAM_CHANNEL` | username канала без `@` — что зеркалим |
| Импорт | `TELEGRAM_API_ID/API_HASH/PHONE` | вход под личным аккаунтом для `make sync`, см. [sync.md](sync.md) |
| Бот | `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHANNEL_REFRESH_INTERVAL` | живые обновления, см. [bot.md](bot.md) |
| SEO | `OPENROUTER_API_KEY/MODEL/FALLBACK_MODEL` | генерация `seo_title/description/keywords`, см. [worker.md](worker.md) |
| Сеть | `PROXY_URL` | если Telegram недоступен напрямую; используется всем исходящим трафиком |
| Публичные адреса | `PUBLIC_SITE_URL`, `PUBLIC_API_URL` | канонические URL в вёрстке, фидах и CORS |
| Аналитика | `YANDEX_METRIKA_ID` | счётчик Метрики; пусто — счётчик выключен |

## 2. Поднять стек

```bash
make up        # docker compose up -d --build
```

Что происходит под капотом, по порядку:

1. **postgres** стартует и проходит healthcheck
2. **migrate** прогоняет миграции и завершается
3. **minio** поднимается, **minio-init** создаёт бакет и открывает публичное
   чтение (медиа раздаются напрямую из MinIO)
4. **api**, **web**, **bot**, **worker** стартуют параллельно

Проверка:

- Сайт — <http://localhost:3000>
- API — <http://localhost:8080/api/v1/posts>
- Консоль MinIO — <http://localhost:9001>

Блог на этом этапе пустой — это нормально

## 3. Наполнить постами

Два независимых пути, работают и вместе:

- **История канала** — разовый импорт под личным аккаунтом:
  `make sync`, пошагово в [sync.md](sync.md)
- **Новые посты** — дальше блог обновляется сам: бот ловит публикации,
  правки и удаления, [bot.md](bot.md)

Медиа и SEO-метаданные доезжают асинхронно — их разбирает
[воркер](worker.md), публикация постов их не ждёт.

Обновление на сервере:

```bash
git pull && make up      # пересоберёт изменившиеся образы и перезапустит
```

## Обслуживание

| Команда | Что делает |
| --- | --- |
| `make logs` / `make ps` | логи и статус всех сервисов |
| `make bot` / `make worker` | логи конкретного демона |
| `make jobs` | фоновые задачи, исчерпавшие попытки |
| `make down` | остановить стек |
| `make clean` | остановить и удалить тома |

Все данные лежат обычными папками в `.docker/data/` — Postgres, MinIO,
сессия Telegram, favicon. Бэкап стека = бэкап этой папки; `make clean` её
**не** трогает, чистится она только вручную
